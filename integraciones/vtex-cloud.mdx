## Resumen

**VTEX IO** (también llamado Store Framework o VTEX Cloud) es la arquitectura moderna de VTEX. Usa un sistema de **apps** y componentes. La integración con Proof tiene 3 pasos:

1. Capturar el `tid` de la URL y guardarlo en una cookie.
2. Adjuntar ese valor a la orden de VTEX durante el checkout.
3. Enviar un postback S2S a Proof cuando se completa la orden.

<Info>
  Tu technical account manager te guía en todo el proceso de integración. Los ejemplos de código en esta página son referencia para tu equipo técnico.
</Info>

## Prerrequisitos

- Proyecto VTEX IO configurado con acceso al CLI (`vtex` CLI)
- Acceso al **admin de VTEX** (para configurar notificaciones de órdenes)
- Una cuenta de marca activa en Proof
- Al menos una oferta creada en Proof

---

## Paso 1: Capturar el Transaction ID

Cuando un usuario hace clic en un tracking link, Proof genera un `tid` único y redirige al usuario a tu tienda con ese parámetro en la URL:

```
https://tutienda.com/landing?tid=abc123xyz789
```

<Warning>
  El `tid` solo existe en la URL de la **primera página** en la que aterriza el usuario. Si navega a cualquier otra página, el parámetro se pierde de la URL. Por eso debes capturarlo inmediatamente y guardarlo en una cookie.
</Warning>

### El script de captura

```html
<script>
  // Capturar el Transaction ID del tracking link y guardarlo en una cookie
  var params = new URLSearchParams(window.location.search);
  var efTid = params.get('tid');
  if (efTid) {
    document.cookie = 'proof_tid=' + efTid + '; path=/; max-age=' + (30 * 24 * 60 * 60);
  }
</script>
```

Este script se ejecuta en cada página. Si el usuario llegó desde un tracking link, captura el Transaction ID y lo guarda en una cookie `proof_tid` con expiración de 30 días.

### Crear la Pixel App

Crea una Pixel App de VTEX IO. Es un componente que se ejecuta en todas las páginas.

```bash
vtex init
# Selecciona "pixel" como tipo de app
```

En `manifest.json`:

```json
{
  "name": "proof-tracking",
  "vendor": "tuvendor",
  "version": "0.1.0",
  "builders": {
    "pixel": "0.x"
  }
}
```

Agrega el script de captura en `pixel/head.html`.

```bash
# Desarrollo local
vtex link

# Publicar
vtex publish
vtex install tuvendor.proof-tracking@0.1.0
```

### Verificar que funciona

1. Abre tu tienda en modo incógnito con `?tid=test123` en la URL.
2. Abre **DevTools** > **Application** > **Cookies**.
3. Confirma que existe una cookie `proof_tid` con valor `test123`.
4. Navega a otra página de tu tienda y verifica que la cookie sigue presente.

## Paso 2: Guardar el Transaction ID en la orden

Para que el postback funcione, el Transaction ID debe quedar guardado en los datos de la orden de VTEX. Esto se hace actualizando el `marketingData` del `orderForm` durante el checkout.

En la Pixel App o en un componente del checkout, lee la cookie y pásala al `orderForm` antes de que el usuario complete la compra:

```javascript
// Leer el Transaction ID de la cookie
var match = document.cookie.match(/(?:^|; )proof_tid=([^;]+)/);
var efTid = match ? match[1] : null;

if (efTid) {
  // Adjuntarlo al marketingData de la orden
  fetch('/api/checkout/pub/orderForm/' + orderFormId + '/attachments/marketingData', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      utmSource: 'proof',
      utmCampaign: efTid
    })
  });
}
```

### Sobre el `orderFormId`

La forma de obtener el `orderFormId` puede variar según tu configuración de VTEX IO. Estas son las opciones más comunes:

| Método | Cuándo aplica |
|---|---|
| `vtexjs.checkout.orderFormId` | VTEX Legacy y algunos setups de IO con `vtexjs` disponible |
| `window.__RUNTIME__.orderFormId` | Algunos setups de VTEX IO que exponen el runtime |
| `POST /api/checkout/pub/orderForm` | Universal. Crea o recupera el orderForm activo |

<Warning>
  Tu technical account manager te ayuda a identificar cuál método funciona en tu checkout específico.
</Warning>

### Cuándo ejecutar este código

Este código debe ejecutarse durante el checkout, **antes** de que el usuario complete la compra. El momento exacto depende de tu flujo de checkout. Lo importante es que el `marketingData` quede guardado en el `orderForm` antes de que se cree la orden.

## Paso 3: Reportar la conversión (Postback S2S)

Cuando se completa una orden, tu servidor envía la conversión a Proof. El postback es una solicitud **GET** con parámetros en la URL.

<Warning>
  El postback es un **GET request**, no un POST. Los parámetros van en la URL, no en el body.
</Warning>

### Configurar el webhook de VTEX

1. Ve a **VTEX Admin** > **Orders Management** > **Settings** > **Notification**.
2. Configura un endpoint en tu servidor que reciba la notificación de órdenes nuevas.

### URL del postback

<Info>
  Tu technical account manager te proporciona la URL exacta del postback para tu integración. La URL del ejemplo a continuación es solo de referencia.
</Info>

### Parámetros

| Parámetro | Requerido | Descripción |
|---|---|---|
| `nid` | Sí | ID de tu red en Proof. Tu technical account manager te lo proporciona. |
| `transaction_id` | Sí | El Transaction ID capturado en el Paso 1. Se extrae de `marketingData.utmCampaign` de la orden. |
| `amount` | Depende | Requerido si tu oferta usa modelo **CPS** (porcentaje sobre la venta). Para ofertas **CPA** (monto fijo por conversión) no es necesario. |
| `adv_event_id` | Recomendado | Tu número de orden en VTEX. Proof lo guarda para reconciliación. |

### Disparar el postback desde tu servidor

```javascript
// Cuando VTEX notifica una nueva orden, extraer el Transaction ID y disparar el postback
const efTid = order.marketingData.utmCampaign;
const orderTotal = order.value / 100; // VTEX guarda los montos en centavos

const params = new URLSearchParams({
  nid: '3935',
  transaction_id: efTid,
  amount: String(orderTotal),
  adv_event_id: order.orderId
});

// GET request a la URL de postback de Proof
await fetch(`https://www.track.proof.nilho.com/?${params}`);
```

<Warning>
  VTEX almacena los montos en **centavos**. Un pedido de $1,500.00 se almacena como `150000`. Divide por 100 antes de enviar el `amount` a Proof.
</Warning>

### Protección contra duplicados

Proof rechaza automáticamente valores de `transaction_id` duplicados. Esto significa que puedes implementar reintentos en tu servidor si el primer intento falla, sin preocuparte por generar conversiones dobles.

## Probar la integración

<Steps>
  <Step title="Verifica la captura del Transaction ID">
    Abre tu tienda con `?tid=test123` en la URL. Verifica en **DevTools** > **Application** > **Cookies** que la cookie `proof_tid` existe con el valor correcto. Navega a otras páginas y confirma que la cookie persiste.
  </Step>
  <Step title="Genera un tracking link de prueba en Proof">
    Crea un tracking link con un creador de prueba para tu oferta.
  </Step>
  <Step title="Haz clic en el tracking link">
    Abre el link en un navegador en modo incógnito. Verifica que llegas a tu tienda VTEX y que la cookie `proof_tid` se crea.
  </Step>
  <Step title="Completa una compra de prueba">
    Completa el checkout con un producto de prueba. Verifica que el Transaction ID queda en `marketingData.utmCampaign` de la orden.
  </Step>
  <Step title="Verifica en Proof">
    En **Reports** > **Conversions**, confirma que la conversión aparece con el monto y número de orden correctos.
  </Step>
</Steps>

## Solución de problemas

| Problema | Qué verificar |
|---|---|
| La cookie `proof_tid` no se crea | Confirma que el script de captura se ejecuta en todas las páginas. Revisa que la URL contiene `tid` como parámetro. |
| El `tid` no aparece en la URL | El usuario no llegó desde un tracking link de Proof. Genera un tracking link de prueba en tu dashboard. |
| Las conversiones no aparecen en Proof | 1. Verifica que el webhook de VTEX está apuntando a tu servidor. 2. Confirma que `marketingData.utmCampaign` tiene el Transaction ID. 3. Revisa que el postback es un GET (no POST). 4. Revisa los logs de tu servidor. |
| El monto no coincide | VTEX almacena montos en centavos. Divide por 100 antes de enviar a Proof. |
| Proof rechaza la conversión como duplicada | Es el comportamiento esperado. Cada `transaction_id` solo puede tener una conversión. |

<Accordion title="La Pixel App no se ejecuta">
  Verifica que la app está correctamente instalada con `vtex list`. Asegúrate de que el `manifest.json` tiene los builders correctos.
</Accordion>
