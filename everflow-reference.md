# Everflow Platform Reference (Internal)

> This file is for documentation authors. It captures Everflow concepts, terminology, and integration details to ensure accuracy when writing advertiser-facing docs for Proof by Nilho.

---

## 1. Core Concepts

### Roles

- **Advertiser**: The brand or merchant running offers. They own the products/services being promoted. In Proof, this is the primary audience.
- **Partner (Affiliate)**: The publisher, influencer, or traffic source that promotes the advertiser's offers in exchange for a commission.
- **Network**: The intermediary platform (Proof/Everflow) that connects advertisers with partners, handles tracking, attribution, and payouts.

### Offer

An **Offer** is the central unit in Everflow. It represents a promotional campaign that an advertiser creates for partners to promote.

Components of an Offer:
- **Offer Name**: Descriptive name.
- **Offer URL (Destination URL)**: The landing page where users are sent after clicking a tracking link.
- **Tracking Domain**: The domain used for click tracking URLs (e.g., `track.proof.nilho.com`).
- **Tracking Link**: Generated URL that partners use. Format: `https://{tracking_domain}/click?oid={offer_id}&affid={partner_id}`
- **Revenue Model**: How the advertiser pays (CPA, CPS, CPC, CPM, Revshare, etc.).
  - **CPA** (Cost Per Action): Fixed amount per conversion.
  - **CPS** (Cost Per Sale): Percentage of sale amount.
  - **CPC** (Cost Per Click): Payment per click.
  - **Revshare**: Revenue share percentage.
- **Payout**: What the partner earns per conversion (can differ from revenue).
- **Conversion Events**: What counts as a conversion (sale, lead, install, etc.).
- **Offer Status**: Active, Paused, Expired, Pending.

### Conversion Events

An offer can have multiple conversion events:
- **Default Event**: The primary conversion (e.g., a sale).
- **Custom Events**: Additional tracked actions (e.g., add to cart, registration, checkout started).

Each event has:
- **Event ID**: Unique numeric identifier.
- **Event Name**: Descriptive name.
- **Revenue/Payout**: Can be set per event.

### Attribution

- **Click ID (`_ef_transaction_id`)**: A unique identifier generated by Everflow for each click. This is the core tracking parameter. When a user clicks a tracking link, Everflow generates a `transaction_id` and appends it to the destination URL.
- **Transaction ID (`transaction_id`)**: The click ID that ties a click to a conversion. Often used interchangeably with Click ID.
- **Order ID (`adv_event_id`)**: The advertiser's own order/transaction identifier. Used for reconciliation.
- **Coupon Code (`coupon_code`)**: For coupon-based attribution where no click is required.

### Attribution Models

1. **Last Click (default)**: Conversion credited to the last partner whose link was clicked.
2. **First Click**: Conversion credited to the first partner whose link was clicked.
3. **Coupon/Promo Code**: Attribution based on a coupon code regardless of click history.
4. **Email Attribution**: Attribution via email identifier.

### Attribution Window

The time period during which a click can be credited for a conversion. Default is typically 30 days. Configurable per offer.

## 2. Tracking Flow

### Click Tracking

1. User clicks a partner's tracking link.
2. Request hits Everflow's tracking domain.
3. Everflow logs the click, generates a `transaction_id`.
4. User is redirected to the offer URL with `_ef_transaction_id={transaction_id}` appended.
5. The landing page should capture and store this parameter (via cookie or URL parameter propagation).

### Conversion Tracking

When the desired action occurs (e.g., purchase), the conversion must be reported back to Everflow.

Three methods:

#### Method 1: Server-to-Server (S2S) Postback — Recommended

The advertiser's server fires an HTTP request to Everflow when a conversion happens.

```
https://www.{tracking_domain}/sdk/conversion?verification_token={token}&transaction_id={click_id}&amount={sale_amount}&adv_event_id={order_id}&offer_id={offer_id}
```

**Parameters:**
- `verification_token`: Advertiser's API verification token from Everflow.
- `transaction_id`: The `_ef_transaction_id` from the click.
- `amount`: Sale amount (for CPS).
- `adv_event_id`: Advertiser's order ID for reconciliation.
- `offer_id`: The Everflow offer ID.
- `coupon_code`: (Optional) For coupon attribution.
- `event_id`: (Optional) For custom events, the numeric event ID.

**Pros:** Most reliable, server-side, not affected by ad blockers or cookie restrictions.
**Cons:** Requires server-side development.

#### Method 2: JavaScript SDK

Include Everflow's JavaScript SDK on the conversion page (e.g., order confirmation page).

```html
<script src="https://www.{tracking_domain}/scripts/sdk/everflow.js"></script>
<script>
  EF.click({
    offer_id: EF.urlParameter('oid'),
    affiliate_id: EF.urlParameter('affid'),
    transaction_id: EF.urlParameter('_ef_transaction_id')
  });
</script>
```

For conversion firing:
```html
<script src="https://www.{tracking_domain}/scripts/sdk/everflow.js"></script>
<script>
  EF.conversion({
    offer_id: {offer_id},
    amount: {sale_amount},
    adv_event_id: '{order_id}',
    verification_token: '{token}'
  });
</script>
```

The SDK automatically reads `_ef_transaction_id` from cookies if present.

**Pros:** Easy to implement, handles cookie management.
**Cons:** Can be blocked by ad blockers or browser privacy settings.

#### Method 3: HTML Pixel (Image Pixel)

Place an invisible image tag on the conversion page.

```html
<img src="https://www.{tracking_domain}/sdk/conversion?verification_token={token}&transaction_id={click_id}&amount={sale_amount}&adv_event_id={order_id}&offer_id={offer_id}" width="1" height="1" />
```

**Pros:** Simplest implementation.
**Cons:** Least reliable, blocked by many browsers, not recommended.

## 3. Offer Configuration (Step by Step)

### General Settings
- Offer Name
- Offer Status (Active/Paused)
- Category
- Offer URL (destination landing page)
- Preview URL (for partner preview, optional)
- Offer Description

### Revenue & Payout
- Revenue Type: CPA / CPS / CPC / Revshare
- Revenue Amount or Percentage
- Payout Type: Same options
- Payout Amount or Percentage
- Can configure different payouts per partner

### Attribution Settings
- Attribution Model: Last Click / First Click
- Click-to-Conversion Window (days)
- Enable Coupon Attribution (yes/no)
- Unique Click ID Conversion (prevent duplicate conversions per click)

### Tracking Settings
- Tracking Domain selection
- Destination URL parameters configuration
- Append click parameters to URL (auto or manual)

### Controls
- Allowed Countries (geo targeting)
- Cap Settings (daily/weekly/monthly conversion caps)
- Scheduling (start/end dates)
- Visibility (public marketplace or private)

## 4. Platform Integrations

### Shopify Integration

Everflow provides a native Shopify app for seamless integration.

**Setup Steps:**
1. In Everflow, go to Advertisers > Integrations > Shopify.
2. Click "Install Shopify App" — redirects to Shopify App Store.
3. Install the Everflow Tracking app in your Shopify store.
4. Authorize the connection between Everflow and Shopify.
5. Configure the integration settings in Everflow:
   - Select which Offer to track.
   - Enable automatic click tracking (injects script on all pages).
   - Enable automatic conversion tracking (fires on order confirmation).

**How it works:**
- The app injects Everflow's JavaScript SDK on all store pages.
- When a user arrives via a tracking link, the SDK captures the `_ef_transaction_id` and stores it in a cookie.
- On purchase, the app automatically fires a conversion with:
  - `transaction_id` from the stored cookie
  - `amount` from the Shopify order total
  - `adv_event_id` as the Shopify order number
  - `order_id` for reconciliation

**Reconciliation:**
- Shopify's Cart ID maps to Everflow's Transaction ID.
- The Shopify order number is passed as `adv_event_id`.
- Refunds in Shopify are not automatically synced — use Everflow's API or manual adjustment.

**Organic Tracking:**
- The Everflow SDK can track organic (non-affiliate) conversions.
- These appear in reports but are not attributed to any partner.
- Useful for understanding overall conversion volume.

**Additional Events:**
- Add to Cart
- Checkout Started/Initiated
- Page View
- These require custom event configuration in the Everflow offer.

**Testing:**
1. Use a test tracking link with a test partner.
2. Click the link, complete a test purchase.
3. Verify in Everflow that the conversion appears with correct amount and order ID.
4. Check the click-to-conversion path in reporting.

### VTEX Integration

Everflow does NOT have a native VTEX integration. This must be implemented manually using the tracking methods described above.

#### VTEX Platform Types

VTEX has three main platform variants:

1. **VTEX Legacy (Portal/CMS)**: The original monolithic VTEX platform. Uses server-rendered templates. Checkout is managed by VTEX's classic checkout system.

2. **VTEX IO / Store Framework (Cloud)**: Modern VTEX architecture. Uses a component-based approach with VTEX IO apps. The store runs on VTEX's cloud infrastructure.

3. **VTEX Headless (FastStore / Custom Frontend)**: Decoupled architecture where the frontend is a separate application (often React/Next.js via FastStore). Communicates with VTEX via APIs.

#### VTEX Legacy Integration

**Click Tracking:**
- Add the Everflow JavaScript SDK to the store's header template (via VTEX CMS).
- On the landing page, capture `_ef_transaction_id` from the URL and store in a cookie.

```html
<!-- In VTEX CMS header template -->
<script src="https://www.{tracking_domain}/scripts/sdk/everflow.js"></script>
<script>
  EF.click({
    offer_id: EF.urlParameter('oid'),
    affiliate_id: EF.urlParameter('affid'),
    transaction_id: EF.urlParameter('_ef_transaction_id')
  });
</script>
```

**Conversion Tracking (JavaScript SDK on Order Confirmation):**
- VTEX Legacy exposes order data on the order confirmation page via the `orderPlaced` event or `dataLayer`.
- Add conversion script to the checkout confirmation template:

```html
<script src="https://www.{tracking_domain}/scripts/sdk/everflow.js"></script>
<script>
  // VTEX exposes order data in the dataLayer or via OMS API
  var orderData = /* extract from VTEX dataLayer or page variables */;
  EF.conversion({
    offer_id: {offer_id},
    amount: orderData.transactionTotal,
    adv_event_id: orderData.transactionId,
    verification_token: '{token}'
  });
</script>
```

**Alternative — S2S Postback:**
- Use VTEX's webhook/notification system (OMS hook) to fire a server-side postback to Everflow on order placement.
- Requires storing the `_ef_transaction_id` in the order's custom data or `marketingData` field.

#### VTEX IO / Cloud Integration

**Click Tracking:**
- Create a custom VTEX IO Pixel App or use the existing Google Tag Manager pixel to inject the Everflow SDK.
- The pixel app runs on every page and can capture/store click parameters.

```javascript
// Inside a VTEX IO Pixel App
import { canUseDOM } from 'vtex.render-runtime'

if (canUseDOM) {
  // Load Everflow SDK
  const script = document.createElement('script')
  script.src = 'https://www.{tracking_domain}/scripts/sdk/everflow.js'
  script.onload = function() {
    EF.click({
      offer_id: EF.urlParameter('oid'),
      affiliate_id: EF.urlParameter('affid'),
      transaction_id: EF.urlParameter('_ef_transaction_id')
    })
  }
  document.head.appendChild(script)
}
```

**Conversion Tracking:**
- The VTEX IO Pixel App receives `order-placed` events with order data.
- Fire the Everflow conversion in the event handler:

```javascript
// In the pixel app's event handler
function handleOrderPlaced(event) {
  const { transactionId, transactionTotal } = event.data
  EF.conversion({
    offer_id: {offer_id},
    amount: transactionTotal,
    adv_event_id: transactionId,
    verification_token: '{token}'
  })
}
```

**Alternative — S2S Postback:**
- Use VTEX's OMS webhooks to fire a server-side postback to Everflow.
- The `_ef_transaction_id` must be stored in `marketingData.utmSource` or custom order fields.

#### VTEX Headless (FastStore / Custom Frontend) Integration

**Click Tracking:**
- Since the frontend is a custom React/Next.js application, directly include the Everflow JavaScript SDK or use S2S tracking.
- On landing page load, read `_ef_transaction_id` from URL params and store in a cookie or localStorage.

```javascript
// In a React/Next.js component or layout
useEffect(() => {
  const params = new URLSearchParams(window.location.search)
  const efTransactionId = params.get('_ef_transaction_id')
  if (efTransactionId) {
    document.cookie = `ef_tid=${efTransactionId}; path=/; max-age=${30*24*60*60}`
  }
}, [])
```

**Conversion Tracking (Client-side):**
```javascript
// After successful checkout, on confirmation page
const efTid = getCookie('ef_tid')
if (efTid) {
  fetch(`https://www.{tracking_domain}/sdk/conversion?verification_token={token}&transaction_id=${efTid}&amount=${orderTotal}&adv_event_id=${orderId}&offer_id={offer_id}`)
}
```

**Conversion Tracking (Server-side — Recommended):**
```javascript
// In API route or server function after order confirmation
const response = await fetch(
  `https://www.{tracking_domain}/sdk/conversion?` +
  `verification_token=${EVERFLOW_TOKEN}` +
  `&transaction_id=${efTransactionId}` +
  `&amount=${orderTotal}` +
  `&adv_event_id=${orderId}` +
  `&offer_id=${offerId}`
)
```

**Pros of S2S in headless:** Full control, reliable, not dependent on client-side.
**Cons:** Requires backend implementation and storing the click ID across the checkout flow.

## 5. API Reference

### Verification Token
Each advertiser has a unique `verification_token` found in:
Everflow Dashboard > Company Settings > Advertiser Settings > Verification Token

### Key API Endpoints

**Fire Conversion:**
```
GET https://www.{tracking_domain}/sdk/conversion
  ?verification_token={token}
  &transaction_id={click_id}
  &amount={amount}
  &adv_event_id={order_id}
  &offer_id={offer_id}
  &coupon_code={coupon}         # optional
  &event_id={event_id}          # optional, for custom events
  &parameter1-5={custom_data}   # optional
```

**Click Tracking URL Format:**
```
https://{tracking_domain}/click?oid={offer_id}&affid={partner_id}&sub1={custom_param}
```

### Sub Parameters
Tracking links support sub-parameters (`sub1` through `sub5`) for custom data passing:
- `sub1`: Often used for campaign name
- `sub2`: Traffic source
- `sub3`: Creative ID
- `sub4-5`: Additional custom data

## 6. Reporting

### Key Reports for Advertisers
- **Conversion Report**: All conversions with click details, partner, amount, order ID.
- **Click Report**: All clicks with timestamps, geo data, device info.
- **Offer Performance**: Summary stats per offer (clicks, conversions, revenue, CR%).
- **Partner Performance**: Performance breakdown by partner.
- **Daily Summary**: Day-by-day performance metrics.

### Key Metrics
- **Clicks**: Total and unique clicks.
- **Conversions**: Total conversions fired.
- **Revenue**: Total revenue (what advertiser pays).
- **Payout**: Total payout (what partners earn).
- **CR% (Conversion Rate)**: Conversions / Clicks.
- **EPC (Earnings Per Click)**: Revenue / Clicks.

## 7. Glossary

| Term | Definition |
|------|-----------|
